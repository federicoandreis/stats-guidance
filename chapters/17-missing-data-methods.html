17  Missing Data: Computational Methods

TipWhat this is for

You have gaps in your data and a decision to make about them – and you need the tools that turn the conceptual framework for missing data into reproducible work. For example:

  • How much is actually missing in this provider return, once the disguised gaps are counted?
  • Which variables go missing together, and in which providers?
  • What does the analysis conclude under complete cases, under imputation, and under honest worst cases – and do those conclusions agree?

This chapter is the machinery: making missingness visible, mapping its structure, handling it in ways that keep the uncertainty honest, and stress-testing whatever you conclude.

17.1 From Reading Gaps to Handling Them

The foundations chapter did the thinking, and this chapter assumes all of it: the three mechanisms and the caution that they are a summary rather than a verdict; the two distinct harms, lost precision and bias; the hard fact that MAR versus MNAR cannot be settled from the data, only argued from evidence; and the regulatory inversion – that in CQC’s work the pattern of who is missing is often a finding in its own right, not an obstacle. What remains is the practical question that chapter deliberately deferred: how, concretely, in code, with the uncertainty kept honest.

One sentence from the foundations bears repeating before any code runs, because every mistake in this chapter’s territory traces back to forgetting it: the goal of handling missing data is valid estimates with honest uncertainty, not accurate predictions of the values that are gone. Lost data is lost. Nothing below recovers it; the methods differ only in how openly they admit that.

17.2 Before You Start

Read the foundations first. The methods here are easy to run and dangerously easy to run on autopilot; the foundations chapter supplies the questions – above all why is this missing? – that decide which method is even admissible.

Data requirements. Any dataset with gaps, which is to say any real dataset. The structured methods (imputation especially) additionally need observed variables that plausibly relate to the missing ones – auxiliary information is the fuel they run on, and without it they idle convincingly.

When these methods are not the point. When missingness is substantial and plausibly tied to the missing values themselves – the struggling providers not reporting – no computation below fixes that; the honest outputs are a sensitivity range and an investigation list, and the foundations chapter’s regulatory inversion is the frame to reach for.

QA framework alignment. Missing-data handling most directly supports QA question 1 (Data – is the data fit for the analysis?) and question 5 (Fitness for Purpose – are limitations declared?). The documentation this chapter ends with is a Code of Practice expectation, not a courtesy.

Worked examples for this chapter. Example 1: GP Access Analysis handles survey missingness inside a z-scoring workflow; the missing-data steps of every other worked example follow the pattern set here.

The code below uses pseudo-randomly generated data. The Python tabs read the R-generated dataset via r.care_homes so both languages show identical numbers; commented-out code shows how to generate the data in Python independently.

17.3 Step 1: Never Trust “0% Missing”

Before counting gaps, make sure the gaps are visible at all. Real datasets disguise missingness as values: a 9999 where a count should be, a 0 that means “not recorded” rather than “none”, an "Unknown" or "N/A" string, a date of 1900-01-01, a blank that one system exports as an empty string and another as the word NULL. A column that reports itself complete may simply have its gaps in costume – and every summary statistic computed over sentinel codes is corrupted in proportion.

The discipline is short and pays for itself every time: check the data dictionary for declared missing-value codes; tabulate each variable’s most frequent values and inspect anything suspiciously round, repeated, or out of range; check types (numbers arriving as text usually mean stowaway strings); and convert everything that means missing into an explicit NA/NaN before any counting starts. Only then does “how much is missing?” have a trustworthy answer.

17.4 Worked Example: Care Home Quality Returns

Scenario: 450 care homes should each have submitted three figures – a staffing rate, a resident satisfaction score, and their latest inspection score – alongside their size and region, which you hold for all of them. The returns have gaps. You need to map the missingness, judge its mechanism, and estimate national mean satisfaction without fooling yourself.

Because the data are simulated, the truth is planted and known: satisfaction genuinely rises with home size, and small homes are far more likely to fail to submit their satisfaction and staffing figures (with a little background missingness everywhere). That combination is the trap this chapter exists for – the missing homes differ systematically from the reporting ones, on a variable that also drives the outcome. We also know the true national mean satisfaction across all 450 homes, so every method below can be marked against it.

Set up the data

  • R
  • Python
# Simulated returns (seed for reproducibility).
# Planted truth: satisfaction rises with size; small homes (< 40 beds) are
# much more likely to fail to submit satisfaction and staffing.
set.seed(123)
n <- 450
size <- sample(20:100, n, replace = TRUE)
satisfaction_full <- 60 + 0.25 * size + rnorm(n, mean = 0, sd = 8)
true_mean <- mean(satisfaction_full)   # the answer the methods must recover

miss_sat   <- (size < 40 & runif(n) < 0.45) | (runif(n) < 0.05)
miss_staff <- (size < 40 & runif(n) < 0.35) | (runif(n) < 0.04)
miss_insp  <- runif(n) < 0.06

care_homes <- data.frame(
  home_id       = sprintf("CH%03d", 1:n),
  size          = size,
  region        = sample(c("North", "Central", "London", "South"), n,
                         replace = TRUE),

  satisfaction  = ifelse(miss_sat, NA, satisfaction_full),

Map the structure

Counting per variable hides the most useful information: which gaps travel together. An upset plot shows, for each combination of missing variables, how many records carry exactly that combination – the natural picture when several variables can each be present or absent.

The non-reporting homes average around 42 beds against 63 for the reporters – a difference far beyond chance. So the missingness is emphatically not MCAR: it leans on size, a variable we hold, which is the MAR signature. The untestable caveat still applies in full: nothing here rules out the gaps also leaning on satisfaction itself (the unhappy small homes staying silent), and that residual doubt is exactly what the sensitivity analysis at the end is for. For now, the working judgement – declared, not hidden – is MAR through size.

Three ways to handle it, marked against the truth

Complete case analysis – analyse the reporters, drop the rest – is the default everyone reaches for and most software performs silently. Under the mechanism we just diagnosed it is predictably biased: the missing homes are small, small homes score lower, so the surviving sample is too cheerful.

Single imputation by the mean – fill every gap with the observed mean – is the classic worse-than-nothing move. It cannot fix the bias (the filled values inherit the cheerful mean), and it actively damages the uncertainty: the sample swells back to 450 with 62 manufactured “observations” of zero variance, so the standard error shrinks while the estimate stays wrong. Confidence rises as validity falls.

Multiple imputation does what single imputation pretends to. Rather than one filled-in guess, it draws many plausible completed datasets – each gap filled from a model of the observed relationships (here, satisfaction’s dependence on size) plus random noise reflecting what the model cannot know – analyses each one, and pools the results so the spread between imputations feeds honestly into the final uncertainty. The version below is hand-rolled in a dozen lines so the machinery is visible; the Overthinking box covers the pooling rules and the production-grade tools.

  • R
  • Python
obs <- care_homes$satisfaction[!miss]

# 1. Complete case analysis
cc_mean <- mean(obs)
cc_se   <- sd(obs) / sqrt(length(obs))

# 2. Single imputation by the mean
filled  <- ifelse(miss, mean(obs), care_homes$satisfaction)
si_mean <- mean(filled)
si_se   <- sd(filled) / sqrt(n)

# 3. Multiple imputation (m = 20), using the observed satisfaction-size
#    relationship. Each round: bootstrap the reporters (parameter
#    uncertainty), fit satisfaction ~ size, fill the gaps with prediction
#    + residual noise, and record the completed-data estimate.
set.seed(7)
m <- 20
est <- vr <- numeric(m)
reporters <- care_homes[!miss, ]
for (k in 1:m) {
  boot  <- reporters[sample(nrow(reporters), replace = TRUE), ]
  fit   <- lm(satisfaction ~ size, data = boot)
  sigma <- summary(fit)$sigma
  completed <- care_homes$satisfaction
  completed[miss] <- predict(fit, newdata = care_homes[miss, ]) +
    rnorm(sum(miss), 0, sigma)
  est[k] <- mean(completed)
  vr[k]  <- var(completed) / n
}
# Rubin's rules: total variance = within + (1 + 1/m) * between
mi_mean <- mean(est)
mi_se   <- sqrt(mean(vr) + (1 + 1 / m) * var(est))

results <- data.frame(
  approach = c("Complete case", "Mean imputation", "Multiple imputation"),
  estimate = round(c(cc_mean, si_mean, mi_mean), 2),
  std_error = round(c(cc_se, si_se, mi_se), 3)
)
results

rng = np.random.default_rng(7)

obs = care_homes.loc[~miss, "satisfaction"]
n = len(care_homes)

# 1. Complete case analysis
cc_mean, cc_se = obs.mean(), obs.std(ddof=1) / np.sqrt(len(obs))

# 2. Single imputation by the mean
filled = care_homes["satisfaction"].fillna(obs.mean())
si_mean, si_se = filled.mean(), filled.std(ddof=1) / np.sqrt(n)

# 3. Multiple imputation (m = 20), using the observed satisfaction-size
#    relationship: bootstrap the reporters, fit satisfaction ~ size, fill
#    the gaps with prediction + residual noise, record each estimate.
m = 20
reporters = care_homes.loc[~miss, ["size", "satisfaction"]]
est, vr = np.empty(m), np.empty(m)
for k in range(m):
    boot = reporters.sample(len(reporters), replace=True, random_state=k)
    slope, intercept = np.polyfit(boot["size"], boot["satisfaction"], 1)
    resid = boot["satisfaction"] - (intercept + slope * boot["size"])
    sigma = resid.std(ddof=2)
    completed = care_homes["satisfaction"].copy()
    pred = intercept + slope * care_homes.loc[miss, "size"]
    completed[miss] = pred + rng.normal(0, sigma, miss.sum())
    est[k] = completed.mean()
    vr[k] = completed.var(ddof=1) / n
# Rubin's rules: total variance = within + (1 + 1/m) * between
mi_mean = est.mean()
mi_se = np.sqrt(vr.mean() + (1 + 1 / m) * est.var(ddof=1))

print(pd.DataFrame({
    "approach": ["Complete case", "Mean imputation", "Multiple imputation"],
    "estimate": np.round([cc_mean, si_mean, mi_mean], 2),
    "std_error": np.round([cc_se, si_se, mi_se], 3),
}).to_string(index=False))
           approach  estimate  std_error
      Complete case     75.41      0.486
    Mean imputation     75.41      0.419
Multiple imputation     74.61      0.492
print(f"True mean: {true_mean:.2f}")

Read the comparison the way a reviewer would. Complete-case analysis is biased upward by about a point, exactly as the mechanism predicted – and with this much data its interval barely stretches to the truth. Mean imputation keeps the bias and adds false precision; it is the only approach whose interval excludes the right answer, which is the quiet scandal of a method still common in practice. Multiple imputation, fuelled by the size relationship the investigation surfaced, comes within a fifth of a point of the truth with an interval that owns up to the remaining uncertainty. None of this is luck: each result follows from what each method does with information and with doubt.

Stress-test the conclusion

The MAR judgement was an argument, not a proof, so the last step bounds what the unverifiable part could do. Suppose the silent homes are not merely small but unhappy – that their satisfaction runs a full residual standard deviation (8 points) below what the size model predicts; or, in the other direction, that they run 8 points above. Recomputing the national mean under both stories spans roughly 74.3 to 76.5 – about a two-point band. If the decision the estimate feeds would change anywhere inside that band, the missingness genuinely matters and the honest report says so; if not, the conclusion is robust to the assumption you could not test. Either way, the tipping point framing – how bad would the silent homes have to be before the conclusion flips? – converts an untestable assumption into a question a domain expert can actually answer.

Note🤔 Rethinking: imputation is not prediction

The word “imputation” invites a misunderstanding that quietly wrecks analyses: that the job is to predict the missing values, and that a better predictor therefore means better handling. It is not, and it does not.

If prediction were the goal, the ideal imputation would fill each gap with its single most likely value – and that is precisely what single imputation does, and precisely why it fails. Filling every gap with the best guess produces a dataset that is individually plausible and collectively false: too smooth, too certain, its variability flattened and its standard errors shrunk, as the worked example shows. The deliberately noisy fills of multiple imputation – each one a worse prediction than the model’s best guess – produce the estimate that is actually right, with uncertainty that is actually honest. Worse predictions, better statistics: that is not a paradox, it is the whole point. The target was never the individual values (lost data is lost); it was valid estimates of the quantities the analysis is about, with intervals a reader can trust.

The regulatory translation is direct. An imputed satisfaction score for a silent care home is not evidence about that home – it would be indefensible in any conversation about that provider, and no imputation output should ever appear in a provider-level judgement. Imputation exists to stop the national picture being distorted by the silence; what the silence means for the silent home is the other question, and often the more important one.

Warning⚙️ Overthinking: Rubin’s rules, m, and the production tools

Why the pooling works. Each completed dataset yields an estimate \(\hat{q}_k\) and its ordinary variance \(W_k\). Rubin’s rules pool them as \(\bar{q} = \frac{1}{m}\sum \hat{q}_k\) with total variance \(T = \bar{W} + (1 + \tfrac{1}{m})B\), where \(\bar{W}\) is the mean within-imputation variance and \(B\) the variance between the \(m\) estimates. \(B\) is the term single imputation throws away: it measures how much the answer moves when the unknowable values are re-drawn, which is exactly the uncertainty the gaps created. (Significance tests on pooled estimates use a t reference with degrees of freedom that depend on \(B/\bar{W}\) – software handles this.)

Proper imputation needs parameter uncertainty too. Filling gaps with “prediction plus residual noise” from one fitted model still understates \(B\), because it treats the model’s coefficients as known. The bootstrap step in the worked example – refitting the model on a resample before each imputation – is one standard fix; fully Bayesian draws are the other. Library implementations do this correctly by default, which is a reason to prefer them in production.

How many imputations. The old advice of \(m = 5\) dates from when computation was expensive; modern practice scales \(m\) with the fraction of missing information – 20 is a reasonable floor, 50 or more for heavily incomplete data or when the analysis reports variances. The cost is trivial; the stability is not.

The production tools. The hand-rolled loop above is for understanding, not for service. In R, the mice package (van Buuren’s multiple imputation by chained equations) handles many incomplete variables at once, mixed types, and diagnostics; in Python, statsmodels’ MICE implementation and scikit-learn’s IterativeImputer cover similar ground (the latter does not pool – you still own Rubin’s rules). Maximum likelihood (full-information ML) is the equally principled alternative that skips imputation entirely, estimating parameters directly from the observed-data likelihood; for many models it and MI give near-identical answers, and the choice is one of convenience. Chained-equations modelling has real settings to get right – which variables to include (generously: auxiliaries help), how many iterations, convergence checks – so for a high-stakes analysis, involve the Guild rather than accepting defaults.

What none of this touches. Every method in this box assumes MAR given the model’s variables. If the silence depends on the missing values themselves after all that conditioning, MI will be confidently, smoothly wrong – which is why the sensitivity analysis is not an optional extra. Formal MNAR machinery exists (selection models, pattern-mixture models) and rests on assumptions as unverifiable as the problem it addresses; it is Guild territory, and its main practical use is disciplined sensitivity analysis rather than rescue. Survey-style non-response weighting (Laaksonen’s territory) is the other principled MAR route – reweight the reporters to stand for the population – and shares both the strengths and the assumption.

17.5 Documenting Your Analysis

For QA purposes, record: how much was missing, per variable and overall, after sentinel codes were converted; the structure (the upset plot or missingness map, retained as evidence); the mechanism judgement and the evidence behind it – including the comparisons of missing versus complete cases on observed variables, and an honest statement of what could not be tested; the handling chosen, with the model and settings if imputation was used (and the software’s version – defaults change); the sensitivity analysis and the band it produced; and what the missingness means for confidence in the conclusions, in words a non-specialist reader can weigh. This record is what the Code of Practice’s quality pillar looks like in practice – and for any consequential analysis, it is the difference between a result that survives challenge and one that does not.

17.7 When to Seek Support from the Guild

Raise your analysis with the Quantitative Analytics and Statistics Guild (the Guild) at a Quant Clinic session when: the missingness is substantial and plausibly MNAR – especially when non-submission correlates with risk; a multiple-imputation model is needed in production (variable selection, chained-equations settings, diagnostics); several variables are incomplete at once with structure between them; weighting and imputation are both candidates and the choice matters; the analysis model is complex enough (multilevel, time series, latent variables) that missingness interacts with it in non-obvious ways; or a published or actioned conclusion would move within the sensitivity band.

**When these methods are not the point.** When missingness is substantial and plausibly tied to the missing values themselves – the struggling providers not reporting – no computation below fixes that; the honest outputs are a sensitivity range and an investigation list, and [the foundations chapter's regulatory inversion](08-missing-data-foundations.qmd) is the frame to reach for.

**QA framework alignment.** Missing-data handling most directly supports QA question 1 (Data – is the data fit for the analysis?) and question 5 (Fitness for Purpose – are limitations declared?). The documentation this chapter ends with is a Code of Practice expectation, not a courtesy.

**Worked examples for this chapter.** [Example 1: GP Access Analysis](../examples/example-01-complete.qmd) handles survey missingness inside a z-scoring workflow; the missing-data steps of every other worked example follow the pattern set here.

*The code below uses pseudo-randomly generated data. The Python tabs read the R-generated dataset via `r.care_homes` so both languages show identical numbers; commented-out code shows how to generate the data in Python independently.*

## Step 1: Never Trust "0% Missing"

Before counting gaps, make sure the gaps are visible at all. Real datasets disguise missingness as values: a `9999` where a count should be, a `0` that means "not recorded" rather than "none", an `"Unknown"` or `"N/A"` string, a date of `1900-01-01`, a blank that one system exports as an empty string and another as the word `NULL`. A column that reports itself complete may simply have its gaps in costume – and every summary statistic computed over sentinel codes is corrupted in proportion.

The discipline is short and pays for itself every time: check the data dictionary for declared missing-value codes; tabulate each variable's most frequent values and inspect anything suspiciously round, repeated, or out of range; check types (numbers arriving as text usually mean stowaway strings); and convert everything that *means* missing into an explicit `NA`/`NaN` before any counting starts. Only then does "how much is missing?" have a trustworthy answer.

## Worked Example: Care Home Quality Returns

*Scenario*: 450 care homes should each have submitted three figures – a staffing rate, a resident satisfaction score, and their latest inspection score – alongside their size and region, which you hold for all of them. The returns have gaps. You need to map the missingness, judge its mechanism, and estimate national mean satisfaction without fooling yourself.

Because the data are simulated, the truth is planted and known: satisfaction genuinely rises with home size, and *small homes are far more likely to fail to submit* their satisfaction and staffing figures (with a little background missingness everywhere). That combination is the trap this chapter exists for – the missing homes differ systematically from the reporting ones, on a variable that also drives the outcome. We also know the true national mean satisfaction across all 450 homes, so every method below can be marked against it.

### Set up the data

# much more likely to fail to submit satisfaction and staffing.

miss_sat   <- (size < 40 & runif(n) < 0.45) | (runif(n) < 0.05)
miss_staff <- (size < 40 & runif(n) < 0.35) | (runif(n) < 0.04)
miss_insp  <- runif(n) < 0.06

care_homes <- data.frame(
  home_id       = sprintf("CH%03d", 1:n),
  size          = size,
  region        = sample(c("North", "Central", "London", "South"), n,
                         replace = TRUE),
  staffing_rate = ifelse(miss_staff, NA, rnorm(n, 3.2, 0.5)),
  satisfaction  = ifelse(miss_sat, NA, satisfaction_full),
  inspection    = ifelse(miss_insp, NA, rnorm(n, 78, 7))
)

colMeans(is.na(care_homes[, c("staffing_rate", "satisfaction", "inspection")]))
cat("True mean satisfaction (all 450 homes):", round(true_mean, 2), "\n")

### Interrogate the mechanism

The gaps cluster; the question is whether they cluster along variables you *hold*. Compare what you know about the missing homes with the complete ones:

:::: {.panel-tabset group="language"}
## R
```{r}
#| label: missing-mechanism
miss <- is.na(care_homes$satisfaction)

aggregate(size ~ miss, data = transform(care_homes, miss = miss), mean)

# Welch two-sample comparison of home size by missingness status
t.test(care_homes$size[miss], care_homes$size[!miss])
```

## Python
```{python}
#| label: missing-mechanism-py
from scipy import stats

miss = care_homes["satisfaction"].isna()

print(care_homes.groupby(miss)["size"].agg(["mean", "count"]).round(1))

# Welch two-sample comparison of home size by missingness status
res = stats.ttest_ind(care_homes.loc[miss, "size"],
                      care_homes.loc[~miss, "size"], equal_var=False)
print(f"t = {res.statistic:.2f}, p = {res.pvalue:.2e}")
```
::::

The non-reporting homes average around 42 beds against 63 for the reporters – a difference far beyond chance. So the missingness is emphatically not MCAR: it leans on size, a variable we hold, which is the MAR signature. The [untestable caveat](08-missing-data-foundations.qmd) still applies in full: nothing here rules out the gaps *also* leaning on satisfaction itself (the unhappy small homes staying silent), and that residual doubt is exactly what the sensitivity analysis at the end is for. For now, the working judgement – declared, not hidden – is MAR through size.

### Three ways to handle it, marked against the truth

**Complete case analysis** – analyse the reporters, drop the rest – is the default everyone reaches for and most software performs silently. Under the mechanism we just diagnosed it is *predictably biased*: the missing homes are small, small homes score lower, so the surviving sample is too cheerful.

**Single imputation by the mean** – fill every gap with the observed mean – is the classic worse-than-nothing move. It cannot fix the bias (the filled values inherit the cheerful mean), and it actively damages the uncertainty: the sample swells back to 450 with 62 manufactured "observations" of zero variance, so the standard error *shrinks* while the estimate stays wrong. Confidence rises as validity falls.

**Multiple imputation** does what single imputation pretends to. Rather than one filled-in guess, it draws *many* plausible completed datasets – each gap filled from a model of the observed relationships (here, satisfaction's dependence on size) *plus* random noise reflecting what the model cannot know – analyses each one, and pools the results so the spread *between* imputations feeds honestly into the final uncertainty. The version below is hand-rolled in a dozen lines so the machinery is visible; the Overthinking box covers the pooling rules and the production-grade tools.

:::: {.panel-tabset group="language"}
## R
```{r}
#| label: missing-handle
obs <- care_homes$satisfaction[!miss]

# 1. Complete case analysis
cc_mean <- mean(obs)
cc_se   <- sd(obs) / sqrt(length(obs))

# 2. Single imputation by the mean
filled  <- ifelse(miss, mean(obs), care_homes$satisfaction)
si_mean <- mean(filled)
si_se   <- sd(filled) / sqrt(n)

# 3. Multiple imputation (m = 20), using the observed satisfaction-size
#    relationship. Each round: bootstrap the reporters (parameter
#    uncertainty), fit satisfaction ~ size, fill the gaps with prediction
#    + residual noise, and record the completed-data estimate.
set.seed(7)
m <- 20
est <- vr <- numeric(m)
reporters <- care_homes[!miss, ]
for (k in 1:m) {
  boot  <- reporters[sample(nrow(reporters), replace = TRUE), ]
  fit   <- lm(satisfaction ~ size, data = boot)
  sigma <- summary(fit)$sigma
  completed <- care_homes$satisfaction
  completed[miss] <- predict(fit, newdata = care_homes[miss, ]) +
    rnorm(sum(miss), 0, sigma)
  est[k] <- mean(completed)
  vr[k]  <- var(completed) / n
}
# Rubin's rules: total variance = within + (1 + 1/m) * between
mi_mean <- mean(est)
mi_se   <- sqrt(mean(vr) + (1 + 1 / m) * var(est))

results <- data.frame(
  approach = c("Complete case", "Mean imputation", "Multiple imputation"),
  estimate = round(c(cc_mean, si_mean, mi_mean), 2),
  std_error = round(c(cc_se, si_se, mi_se), 3)
)
results
cat("True mean:", round(true_mean, 2), "\n")
```

## Python
```{python}
#| label: missing-handle-py
rng = np.random.default_rng(7)

obs = care_homes.loc[~miss, "satisfaction"]
n = len(care_homes)

# 1. Complete case analysis
cc_mean, cc_se = obs.mean(), obs.std(ddof=1) / np.sqrt(len(obs))

# 2. Single imputation by the mean
filled = care_homes["satisfaction"].fillna(obs.mean())
si_mean, si_se = filled.mean(), filled.std(ddof=1) / np.sqrt(n)

# 3. Multiple imputation (m = 20), using the observed satisfaction-size
#    relationship: bootstrap the reporters, fit satisfaction ~ size, fill
#    the gaps with prediction + residual noise, record each estimate.
m = 20
reporters = care_homes.loc[~miss, ["size", "satisfaction"]]
est, vr = np.empty(m), np.empty(m)
for k in range(m):
    boot = reporters.sample(len(reporters), replace=True, random_state=k)
    slope, intercept = np.polyfit(boot["size"], boot["satisfaction"], 1)
    resid = boot["satisfaction"] - (intercept + slope * boot["size"])
    sigma = resid.std(ddof=2)
    completed = care_homes["satisfaction"].copy()
    pred = intercept + slope * care_homes.loc[miss, "size"]
    completed[miss] = pred + rng.normal(0, sigma, miss.sum())
    est[k] = completed.mean()
    vr[k] = completed.var(ddof=1) / n
# Rubin's rules: total variance = within + (1 + 1/m) * between
mi_mean = est.mean()
mi_se = np.sqrt(vr.mean() + (1 + 1 / m) * est.var(ddof=1))

print(pd.DataFrame({
    "approach": ["Complete case", "Mean imputation", "Multiple imputation"],
    "estimate": np.round([cc_mean, si_mean, mi_mean], 2),
    "std_error": np.round([cc_se, si_se, mi_se], 3),
}).to_string(index=False))
print(f"True mean: {true_mean:.2f}")
```

### Stress-test the conclusion

The MAR judgement was an argument, not a proof, so the last step bounds what the unverifiable part could do. Suppose the silent homes are not merely small but *unhappy* – that their satisfaction runs a full residual standard deviation (8 points) below what the size model predicts; or, in the other direction, that they run 8 points above. Recomputing the national mean under both stories spans roughly 74.3 to 76.5 – about a two-point band. If the decision the estimate feeds would change anywhere inside that band, the missingness genuinely matters and the honest report says so; if not, the conclusion is robust to the assumption you could not test. Either way, the *tipping point* framing – how bad would the silent homes have to be before the conclusion flips? – converts an untestable assumption into a question a domain expert can actually answer.

## 🤔 Rethinking: imputation is not prediction

The word "imputation" invites a misunderstanding that quietly wrecks analyses: that the job is to *predict the missing values*, and that a better predictor therefore means better handling. It is not, and it does not.

If prediction were the goal, the ideal imputation would fill each gap with its single most likely value – and that is precisely what single imputation does, and precisely why it fails. Filling every gap with the best guess produces a dataset that is individually plausible and collectively false: too smooth, too certain, its variability flattened and its standard errors shrunk, as the worked example shows. The deliberately *noisy* fills of multiple imputation – each one a worse prediction than the model's best guess – produce the estimate that is actually right, with uncertainty that is actually honest. Worse predictions, better statistics: that is not a paradox, it is the whole point. The target was never the individual values (lost data is lost); it was valid estimates of the quantities the analysis is about, with intervals a reader can trust.

The regulatory translation is direct. An imputed satisfaction score for a silent care home is not evidence about that home – it would be indefensible in any conversation about that provider, and no imputation output should ever appear in a provider-level judgement. Imputation exists to stop the *national* picture being distorted by the silence; what the silence means for the silent home is [the other question, and often the more important one](08-missing-data-foundations.qmd).
:::

## ⚙️ Overthinking: Rubin's rules, m, and the production tools

**Why the pooling works.** Each completed dataset yields an estimate $\hat{q}_k$ and its ordinary variance $W_k$. Rubin's rules pool them as $\bar{q} = \frac{1}{m}\sum \hat{q}_k$ with total variance $T = \bar{W} + (1 + \tfrac{1}{m})B$, where $\bar{W}$ is the mean within-imputation variance and $B$ the variance *between* the $m$ estimates. $B$ is the term single imputation throws away: it measures how much the answer moves when the unknowable values are re-drawn, which is exactly the uncertainty the gaps created. (Significance tests on pooled estimates use a t reference with degrees of freedom that depend on $B/\bar{W}$ – software handles this.)

**Proper imputation needs parameter uncertainty too.** Filling gaps with "prediction plus residual noise" from one fitted model still understates $B$, because it treats the model's coefficients as known. The bootstrap step in the worked example – refitting the model on a resample before each imputation – is one standard fix; fully Bayesian draws are the other. Library implementations do this correctly by default, which is a reason to prefer them in production.

**How many imputations.** The old advice of $m = 5$ dates from when computation was expensive; modern practice scales $m$ with the fraction of missing information – 20 is a reasonable floor, 50 or more for heavily incomplete data or when the analysis reports variances. The cost is trivial; the stability is not.

**The production tools.** The hand-rolled loop above is for understanding, not for service. In R, the `mice` package (van Buuren's *multiple imputation by chained equations*) handles many incomplete variables at once, mixed types, and diagnostics; in Python, `statsmodels`' MICE implementation and scikit-learn's `IterativeImputer` cover similar ground (the latter does not pool – you still own Rubin's rules). *Maximum likelihood* (full-information ML) is the equally principled alternative that skips imputation entirely, estimating parameters directly from the observed-data likelihood; for many models it and MI give near-identical answers, and the choice is one of convenience. Chained-equations modelling has real settings to get right – which variables to include (generously: auxiliaries help), how many iterations, convergence checks – so for a high-stakes analysis, involve the Guild rather than accepting defaults.

**What none of this touches.** Every method in this box assumes MAR given the model's variables. If the silence depends on the missing values themselves after all that conditioning, MI will be confidently, smoothly wrong – which is why the sensitivity analysis is not an optional extra. Formal MNAR machinery exists (selection models, pattern-mixture models) and rests on assumptions as unverifiable as the problem it addresses; it is Guild territory, and its main practical use is disciplined sensitivity analysis rather than rescue. Survey-style *non-response weighting* (Laaksonen's territory) is the other principled MAR route – reweight the reporters to stand for the population – and shares both the strengths and the assumption.
:::

## Documenting Your Analysis

For QA purposes, record: how much was missing, per variable and overall, *after* sentinel codes were converted; the structure (the upset plot or missingness map, retained as evidence); the mechanism judgement and the evidence behind it – including the comparisons of missing versus complete cases on observed variables, and an honest statement of what could not be tested; the handling chosen, with the model and settings if imputation was used (and the software's version – defaults change); the sensitivity analysis and the band it produced; and what the missingness means for confidence in the conclusions, in words a non-specialist reader can weigh. This record is what the Code of Practice's quality pillar looks like in practice – and for any consequential analysis, it is the difference between a result that survives challenge and one that does not.
